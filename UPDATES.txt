  What was changed and why
  
  The motivating problem: NodePtr held a raw src_ pointer directly into nodePool. This created two failure modes when the tree was mutated:

  1. Pool reallocation (insertions) — move_alloc or any pool resize moved all nodes to a new address, leaving every live src_ dangling.
  2. Deletion — freeing a node left src_ pointing at freed memory. Any subsequent isMember call dereferenced it, producing either a crash or a silent wrong answer.

  The design that solved it:

  src_ was removed from NodePtr entirely. The only field is now p — the owned deep copy. Two new fields were added to Node itself (nodeId, numRemovesSnapshot) so they are automatically inherited by every copy dispatched from a search.

  isMember now runs three checks in order:

  1. treeId — O(1), catches uninitialized NodePtrs and nodes from a different tree instance entirely. This alone fixes the previously failing IS_MEMBER_AFTER_BUILD_DESTROY_BUILD test (rebuild issues a new treeId).
  2. numRemovesSnapshot == tree%numRemoves — O(1), fast path. If no removals have occurred since this copy was dispatched, the node cannot have been deleted and we return true immediately without touching the pool.
  3. nodePool scan for nodeId — O(n), definitive check. Only runs when removals have happened. Scans the live pool entries (all valid memory, never touches freed nodes) and updates the snapshot on the copy so the fast path fires next time.

  rNN_Node now uses target%p as the search centre (the owned deep copy) rather than target%src_, so it never dereferences a pool pointer for the search itself. The excludeTarget identity check switched from associated(src_, src_) to p%nodeId 
  == target%p%nodeId. Both rNN_Node and rNN_Centroid stamp numRemovesSnapshot = this%numRemoves on every dispatched result so the fast path is active from the moment a NodePtr is returned.
